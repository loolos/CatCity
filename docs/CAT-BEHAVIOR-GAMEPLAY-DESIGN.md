# Cat City 玩法升级设计文档（熟悉 + 意外）

> 目标：在保留当前“可规划、可优化”的乐趣前提下，引入“猫咪不完全听话但可被理解”的行为系统，提升操作手感与重复游玩价值。

## 1. 现状代码结构与主要机制梳理

本节基于当前仓库实现进行总结，作为后续改造的约束。

### 1.1 项目与模块结构

- 游戏核心为前端静态 MVP：`GameController` 负责 Build/Sim 生命周期、事件绑定与渲染节奏。  
- 核心回合模拟在 `Simulation`，包括：生成入口出口、障碍、猫咪出生、需求增长、寻路、设施交互与计分。  
- 主要配置集中在 `src/config.js`（格子、设施上限、需求阈值、分数规则、回合长度等）。

### 1.2 当前玩法主循环（玩家视角）

1. 在 Build 阶段放置设施（鱼缸、猫窝、激光笔、隧道）。
2. 启动模拟后，猫按固定节奏出生并移动。
3. 当饥饿/困倦达到阈值，猫会优先寻路到对应设施满足需求。
4. 需求被满足可得分，窗口内交替满足不同需求可获 loop bonus。
5. 猫最终尝试前往固定出口离场，状态差时触发 penalty。

### 1.2.1 关键澄清：本作是“回合制模拟”，不是“时间制模拟”

- 游戏底层逻辑以 **Turn（回合）** 推进：猫咪、设施、需求增长、服务完成、出生与离场都在离散回合上结算。
- 画面上的移动与表现采用“逐渐播放”的动画，仅用于可视化回合结果，不改变规则结算的离散本质。
- 因此设计新玩法时，必须保证所有新系统（指令、偏航、情绪变化、事件触发）都以“每回合判定/更新”为基本单位。
- 可理解为：**动画是连续观感，规则是离散回合**。

### 1.3 当前机制优点

- **确定性强**：种子 + 回合模拟便于调参与复盘。
- **策略清晰**：设施布局与路径效率强相关。
- **反馈明确**：分数与惩罚可解释。

### 1.4 当前“可玩性天花板”

- 猫行为偏功能化，更多像“移动单位”而非有个性的生命体。
- 玩家操作偏“放置前决策”，运行中动态操作与博弈较少。
- 意外事件不足，重开对局的新鲜感主要来自地图与障碍。

## 2. 设计目标（熟悉 + 意外）

### 2.1 设计原则

- **70% 熟悉（可学习）**：玩家可以通过观察逐步掌握猫的倾向。
- **30% 意外（有生命感）**：猫不会完全按系统逻辑执行，会“偏航”。
- **偏航可解释**：每次意外都能被归因为情绪/性格/环境，不做纯随机。

### 2.2 体验目标

- 玩家感受从“我在摆最优路径”升级为“我在管理一个有性格的猫咪城市”。
- 失败/偏离不只带来惩罚，也能转化为剧情、资源或连锁机会。

## 3. 新系统方案总览

本方案采用“低侵入、可分阶段上线”的方式，优先复用现有 `Simulation.tick()` 结构。

### 3.1 系统 A：猫咪个性与情绪引擎（Trait + Mood）

为每只猫新增以下状态：

- `traits`（常量，出生时决定）
  - `obedience`：服从度（影响是否执行指令）
  - `curiosity`：好奇心（更容易被环境吸引）
  - `sociability`：社交性（更容易靠近他猫/NPC 热点）
  - `stubbornness`：执拗（倾向坚持当前目标）
- `mood`（变量，随回合变化）
  - `stress`：压力（强制命令过多会上升）
  - `focus`：专注（受干扰时下降）
  - `affinityToPlayer`：对玩家信任（长期影响执行率）

**设计收益**

- 同一地图布局下，不同猫群会出现不同故事线。
- 玩家长期可通过“低压力管理”培养更高可控度。

### 3.2 系统 B：指令强度（建议 / 安排 / 严格）

将玩家对猫的干预从“绝对命令”改为“强度选择”：

- `suggest`（建议）：低副作用，执行率中等
- `schedule`（安排）：执行率高，轻微压力
- `strict`（严格）：执行率最高，明显累积压力与逆反风险

**核心公式（建议版）**

`P(execute) = BaseByCommand × obedience × affinity × contextFit - distraction + bonus`

其中：

- `BaseByCommand`: 0.60 / 0.82 / 0.95
- `contextFit`: 当前状态与命令匹配度（例如太困时巡逻匹配度低）
- `distraction`: 周边可互动诱因（激光、箱子、热点）
- `bonus`: 最近连续执行奖励（鼓励形成习惯）

### 3.3 系统 C：偏航事件（Detour）

当命令未执行时，猫进入“偏航子状态”：

- `chase_light`（追光）
- `sunbath`（晒太阳发呆）
- `social_visit`（串门互动）
- `box_hide`（钻箱短暂失联）

每类偏航附带 **风险-收益**：

- 可能拖慢主任务
- 但可触发额外收益（临时客流、隐藏加分、特殊事件）

### 3.4 系统 D：软失败转机会（Fail Forward）

任务失败不直接结算损失，而进入补救分支：

- 迟到 → 夜间补班 mini-event
- 打翻道具 → 清理互动并获得“街区关系值”
- 抗命过高 → 安抚互动降低压力并提高 affinity

这样可以把“猫不听话”从挫败转成内容源。

### 3.5 系统 E：可读反馈层（解释偏航原因）

必须增加实时可读反馈，避免随机感：

- 气泡文本：`“被蝴蝶吸引”`、`“现在只想睡觉”`
- 图标反馈：专注下降、压力升高、好奇触发
- HUD 日志：最近 3 条关键行为决策

## 4. 与当前代码的映射（实施抓手）

### 4.1 Simulation 层修改点

建议在 `src/simulation.js` 中新增：

1. **猫实体扩展**（`spawnCat`）
   - 注入 `traits/mood/currentIntent/detourState`。
2. **目标选择前置判定**（`findTarget` 前）
   - 根据命令强度、状态和性格决定“执行 or 偏航”。
3. **tick 中加入 mood 更新**
   - 压力随严格命令增长，休息/安抚下降。
4. **计分系统扩展**
   - 新增“意外正收益”渠道，避免偏航=纯惩罚。

### 4.2 Controller/UI 层修改点

- `src/game-controller.js`
  - 增加“命令强度”UI 状态读写。
  - 在回合推进时记录行为日志，传入 HUD。
- `src/ui.js`
  - 新增状态图标与事件气泡。
  - 增加“行为日志区”与“猫个性卡片”（可选）。

### 4.3 配置层建议

在 `src/config.js` 增加：

- 命令强度系数
- 偏航事件概率上下限
- 情绪衰减/增长速率
- 软失败补救奖励范围

## 5. 分阶段落地计划（建议）

### Phase 1（低风险，1~2 次迭代）

- 加入 `traits`（先只做 `obedience/curiosity` 两项）。
- 实现命令强度三挡 + 执行概率。
- 未执行时仅触发 2 种偏航（追光/发呆）。
- HUD 显示“本回合决策原因”。

**验收标准**

- 相同种子下，改动前后可复现实验差异。
- 玩家可通过强度选择明显影响猫执行率。
- 偏航有解释文本，且出现频率可调。

### Phase 2（体验增强）

- 上线 `stress/focus/affinity`。
- 引入软失败分支事件。
- 增加偏航正收益。

### Phase 3（内容扩展）

- 增加猫 archetype（社牛、宅猫、猎手等）。
- 按街区/建筑类型赋予环境偏航权重。
- 数据埋点驱动调参（偏航频率、挫败率、收益率）。

---

## 6. 平衡与风险控制

### 6.1 关键平衡指标

- 命令执行率（按强度分组）
- 偏航导致的净损失 / 净收益
- 玩家“可解释性”评分（主观问卷）
- 单局重玩率与平均时长

### 6.2 风险与应对

- **风险：太随机导致失控** → 设“保底执行”与偏航上限。
- **风险：信息太多看不懂** → 只显示“最近原因 + 核心图标”。
- **风险：实现复杂度上升** → 先实现最小闭环（Phase 1）。

---

## 7. 建议先做的最小可行版本（MVP of New玩法）

为了最快验证“熟悉 + 意外”是否成立，推荐先做：

1. 命令强度三挡（仅作用于“去目标设施”决策）。
2. 两个 trait：`obedience` + `curiosity`。
3. 两个偏航：`sunbath`（停 1~2 回合）、`chase_light`（偏离 1 格）。
4. 一条 UI 行为原因提示。

若该版本能显著提升“有趣度”且不显著增加挫败，再推进 Phase 2。

---

## 8. 结论

当前 Cat City 的基础策略框架已经成立，且其核心是“离散回合制”而非“时间制”。下一步重点不是增加更多设施，而是让猫成为“半可控、可理解、有性格”的系统主体。通过“命令强度 + 个性情绪 + 可解释偏航 + 软失败分支”，并统一到逐回合判定，可在保持熟悉策略感的同时稳定产生意外内容，提升操作乐趣与叙事感。
