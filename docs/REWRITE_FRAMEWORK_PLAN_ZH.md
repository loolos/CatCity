# CatCity 重写框架方案（基于当前代码能力盘点）

> 目标：按照你刚刚确定的 Game Design 方向，先做“可落地框架重构”，再逐步扩展玩法；本文件先完整记录**当前已实现功能**与**可放置建筑**，作为重写基线。

## 1. 当前代码库功能盘点（As-Is）

### 1.1 核心玩法循环（当前可玩）

1. **Build 阶段**：玩家在棋盘上放置设施。  
2. **Simulation 阶段**：按回合推进，猫出生、移动、增长需求、寻路、使用设施、结算得分。  
3. **结束条件**：到达固定总回合数后结束，显示最终分数。

### 1.2 当前“已实现”系统能力

- **固定网格地图**：当前为 `5x5`。  
- **入口/出口机制**：每局基于种子生成固定 IN/OUT（入口在一侧，出口在对边）。  
- **障碍物生成**：随机数量石头障碍，且保证入口到出口存在可达路径。  
- **猫咪状态与需求**：
  - Hunger（饥饿）
  - Sleepiness（困倦）
  - 两项需求按回合增长
- **需求驱动寻路**：当需求超过阈值，会寻路到对应设施。  
- **设施服务时间**：鱼缸/猫窝有占用与服务回合，存在等待。  
- **激光引导**：猫经过激光格时，有较高概率被引导到指定方向。  
- **隧道方向约束**：隧道不是全向传送，按 horizontal/vertical 方向限制通过。  
- **得分系统**：
  - 满足需求得分
  - 在窗口期内交替满足需求可触发 loop bonus
  - 满足后离场有奖励
  - 状态差离场有惩罚
- **可视化反馈**：
  - 回合、分数、猫数量、状态
  - 入口出口信息、惩罚信息、最近离场日志
  - 移动动画、猫脚印、分数弹字
  - 鱼缸帧动画
- **确定性种子**：输入 seed 后可复现同一局关键生成逻辑。

### 1.3 当前可放置建筑/对象（Build 可操作）

> 下表区分“玩家可放置”与“系统自动生成”。

| 类型 | 代码 ID | 是否玩家可放置 | 上限 | 关键规则 |
|---|---|---:|---:|---|
| Fish Bowl（鱼缸） | `fish` | 是 | 2 | 满足 hunger，存在服务时间与占用 |
| Cat Bed（猫窝） | `bed` | 是 | 2 | 满足 sleepiness，存在服务时间与占用 |
| Laser Pointer（激光笔） | `laser` | 是 | 5 | 可旋转方向，上格猫有概率被引导 |

## 2. 重写目标与约束（To-Be）

### 2.1 重写目标

- 将当前“功能可跑”的 MVP 重构为“**清晰分层 + 易扩展**”架构。
- 保留现有可玩内容（不先砍功能），先做“兼容性重构”。
- 为后续扩展（个性、情绪、命令强度、偏航事件）预留稳定接口。

### 2.2 重写约束

- **回合制结算不改变**：新系统必须挂接在每回合更新中。  
- **可复现性不丢失**：所有随机必须走统一 RNG。  
- **功能迁移顺序**：先等价迁移，再新增功能。  
- **测试先行补齐**：每迁移一层补对应单测。

## 3. 具体重写框架（先搭骨架）

## 3.1 目录结构（建议）

```text
src/
  core/
    engine/                # 回合引擎、调度器
    model/                 # Cat / Facility / Map 数据模型
    rules/                 # 需求增长、移动、设施交互、结算规则
    rng/                   # 随机与可复现
  gameplay/
    systems/
      needs/               # 饥饿/困倦系统
      pathfinding/         # 寻路
      scoring/             # 分数与奖惩
      flow/                # 出生/离场流程
    content/
      facilities/          # 鱼缸/猫窝/激光/隧道规则插件
      map-generators/      # 地图与障碍生成器
  application/
    use-cases/             # start/reset/place/tick 等用例
    state/                 # App 状态管理
  presentation/
    ui/                    # DOM 渲染
    hud/
    animations/
  adapters/
    storage/
    backend/
  tests/
    unit/
    integration/
```

### 3.2 分层职责

- **Engine 层**：只管“每回合执行顺序与生命周期”。
- **Rule/System 层**：纯业务规则，可单测。
- **Application 层**：处理 UI 输入（点格子、开始、重置）与游戏用例。
- **Presentation 层**：纯渲染，不持有核心规则。

### 3.3 Tick 执行管线（标准化）

每回合固定顺序：

1. `spawnPhase`（出生）
2. `needPhase`（需求增长）
3. `servicePhase`（设施服务推进与完成）
4. `decisionPhase`（目标选择）
5. `movePhase`（移动与引导）
6. `interactionPhase`（尝试占用设施/触发事件）
7. `exitPhase`（离场结算）
8. `scorePhase`（积分汇总）
9. `cleanupPhase`（TTL 老化、垃圾回收）

> 后续扩展新玩法时，原则上只“新增 phase 或插入可配置 hook”，避免改旧逻辑。

### 3.4 关键可扩展点（为后续功能预留）

- `BehaviorPolicy`：猫决策策略接口（后续接入 trait/mood/命令强度）。
- `FacilityRule`：建筑规则插件接口（新增建筑不改主循环）。
- `EventBus`：行为日志与 UI 反馈统一出口。
- `BalanceConfig`：所有数值集中配置，可做 A/B 调参。

## 4. 分阶段实施计划

### Phase A：基线冻结与文档化（当前阶段）

- 完成当前功能与建筑盘点（本文件）。
- 锁定现有规则与参数，作为“等价重构验收基准”。

### Phase B：骨架重构（不加新玩法）

- 拆分 Simulation 大文件为：
  - 引擎（tick orchestration）
  - 需求系统
  - 移动系统
  - 设施系统
  - 结算系统
- 保证“同种子、同输入、结果一致（或可解释差异）”。

### Phase C：先接入你设计文档中的最小新玩法

- 命令强度三挡（建议/安排/严格）
- trait 两项（obedience/curiosity）
- 偏航两项（sunbath/chase_light）
- HUD 决策原因一行

### Phase D：继续扩展

- mood/affinity/stress
- 软失败分支
- 更多事件与建筑内容

---

## 5. 验收标准（每阶段）

- **可运行**：构建与本地启动通过。  
- **可复现**：固定 seed 回放一致。  
- **可测试**：关键规则有单元测试覆盖。  
- **可解释**：关键行为有日志与 HUD 反馈。

---

## 6. 下一步执行建议

1. 先把 `Simulation.tick()` 拆分为 Phase 函数（文件内拆分即可，先不跨目录）。
2. 再把 Phase 抽成 `systems/*` 模块。
3. 最后在 `decisionPhase` 注入新设计（命令强度 + trait）。

> 这样做风险最低，能在持续可玩的前提下完成“先框架、后扩展”的目标。
